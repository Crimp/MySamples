Получиться должно так.

[AspNetCompatibilityRequirements(RequirementsMode = AspNetCompatibilityRequirementsMode.Allowed)]

public class CustomWcfSecuredDataServer : WcfSecuredDataServer {

private static SecuredDataServer securedDataServer;

public static void SetSecuredDataServer(SecuredDataServer dataServer) {

securedDataServer = dataServer;

}

public CustomWcfSecuredDataServer() : base(securedDataServer) { }

}

Добавляем файл Global. Настраиваем в нём Security на Application_Start.


protected void Application_Start(object sender, EventArgs e) {

ConnectionString = ConfigurationManager.ConnectionStrings["AzureApplicationServices"].ConnectionString;

CustomAuthenticationServiceHelper customAuthenticationServiceHelper = new CustomAuthenticationServiceHelper();

QueryRequestSecurityStrategyHandler securityProviderHandler = delegate() {

return new SecurityStrategyComplex(typeof(SecuritySystemUser), typeof(SecuritySystemRole), new AuthenticationStandard());

};

SecuredDataServer dataServer = new SecuredDataServer(

ConnectionString, customAuthenticationServiceHelper.XPDictionary, securityProviderHandler);

CustomWcfSecuredDataServer.SetSecuredDataServer(dataServer);

}

Добавляем в проект класс Global

Тут мы будем реализовывать основную часть обработки запросов на сервис с подключением XafSecurity.


Обработку запросов и проверку UserCredentials производим в методе Application_AuthenticateRequest. В момент получения запроса создаётся новый интстанс SecurityStrategyComplex, производится проверка UserCredentials и в случае успеха запрос пропускается дальше. При получении AuthenticationException выдаётся "401 Unauthorized".


В случае прохождения авторизации обработка запроса переходит в Data сервис. Данное приложение использует соединение с только что нами созданным сервисом CustomAuthenticationDataService как для получение данных, так и для построения UI элементов оперируя данными о том что может делать данный пользователь.

Нам также понадобиться форма LogOn. В данном примере имя пользователя и пароль передаются при каждом запросе.



В примере имеется 3 View отображающие данные с сервера это:

1 contact-ListView

2 contact-DetailView

3 contact-EditView


Мы переименовали формы сгенерированные DXTreme дизайнером, под формат более привычный для XAF пользователей.


(В данном примере мы переделали ToolBar под нужды нашего приложения.)


Мы не будем подробно рассматривать детали написания DXTreme приложений, а сразу рассмотрим несколько моментов имеюших непосредственное отношение во взаимодействии Data сервиса с DXTreme клиентом.


В форме LogOn есть вспомогательный вызов сервисной операции IsUserAllowed. При любом запросе к сервису в запросе отсылается информация пользователя UserName and Password. Если сервер на запрос IsUserAllowed возврашает true значит данный юзер прошел аутентификацию. Любой другой результат (ошибка сервера, недоступность сервера, рассматриваются как обратное утверждение).


В случае возвращения результата «True» функцией IsUserAllowed идёт перенаправление на contact-ListView. При запросе данных с сервера в классе EntityStoreContext пред тем как будет отправлен запрос, к запросу добавляется UserName и Password. В случае возврата сервером ошибки 401(Unauthorized) происходит перенаправление на страницу LogOn. Тут есть проблема, в том что при использовании JSONP мы не получим с сервера код ошибки 401 :(




На форме contact-ListView, есть экшен создания новых контактов, по умолчанию права на создание есть только у пользователя Admin.


В форме contact-DetailView есть 2 вспомогательных вызова сервисных функции, это IsGranted и CanReadMembers.


Через метод IsGranted запрашиваем, если у пользователя права на редактирование данного объекта, и если есть то будет отображена кнопка Edit и Delete (В данном примере допускается, что при наличии прав на редактирование объекта, права на удаление тоже есть). Если нет, то будет показан переключатель «UI Level Security» его можно выставить в OFF тогда результат вызова метода IsGranted будет проигнорирован и кнопка Edit будет отображена.




При нажатии кнопки Edit мы переходим на contact-EditView. Тут мы можем изменить отображаемые данные и нажать клавишу Save. В случае если у текущего пользователя есть права на редактирование данного объекта, данные будут сохранены. В случае же если прав нет, а мы перешли сюда используя переключатель «UI Level Security» проигнорировав результат метода IsGranted, при попытке сохранения мы получим ошибку сервера, так как у пользователя нет прав на данную операцию.





Обработка ошибок на сервере и передача их на клиент, происходит в методе HandleException класса CustomAuthenticationDataService.


protected override void HandleException(HandleExceptionArgs args) {

if(args.Exception.GetType() == typeof(ObjectLayerSecurityException)) {

//Forbidden

args.Exception = new DataServiceException(403, args.Exception.Message);

}

base.HandleException(args);

}


Здесь мы видим что в случае возникновения исключения типа ObjectLayerSecurityException оно перехватывается и в замен создаётся DataServiceException. Это необходимо для того чтоб клиент получил внятный код ошибки.  There are mainly three exceptions: DataServiceClientException for client operations (updates/inserts), DataServiceQueryException for querying via LINQ and DataServiceRequestException for requesting via URI.

Любое другое исключение будет передано на клиент как «Internal Server Error» с кодом ошибки 501, что явно недостаточно в некоторых случаях :)
